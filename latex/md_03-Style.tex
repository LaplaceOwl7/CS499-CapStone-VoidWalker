 Consistency is the most important aspect of style. The second most important aspect is following a style that the average C++ programmer is used to reading.

C++ allows for arbitrary-\/length identifier names, so there\textquotesingle{}s no reason to be terse when naming things. Use descriptive names, and be consistent in the style.


\begin{DoxyItemize}
\item {\ttfamily Camel\+Case}
\item {\ttfamily snake\+\_\+case}
\end{DoxyItemize}

are common examples. {\itshape snake\+\_\+case} has the advantage that it can also work with spell checkers, if desired.\hypertarget{md_03-Style_autotoc_md1}{}\doxysection{\texorpdfstring{Establishing A Style Guideline}{Establishing A Style Guideline}}\label{md_03-Style_autotoc_md1}
Whatever style guidelines you establish, be sure to implement a {\ttfamily .clang-\/format} file that specifies the style you expect. While this cannot help with naming, it is particularly important for an open source project to maintain a consistent style.

Every IDE and many editors have support for clang-\/format built in or easily installable with an add-\/in.

\begin{DoxyItemize}
\item Types start with upper case\+: {\ttfamily My\+Class}.
\item Functions and variables start with lower case\+: {\ttfamily my\+Method}.
\item Constants are all upper case\+: {\ttfamily const double PI=3.\+14159265358979323;}.
\end{DoxyItemize}

C++ Standard Library (and other well-\/known C++ libraries like \href{http://www.boost.org/}{\texttt{ Boost}}) use these guidelines\+:


\begin{DoxyItemize}
\item Macro names use upper case with underscores\+: {\ttfamily INT\+\_\+\+MAX}.
\item Template parameter names use Pascal case\+: {\ttfamily Input\+Iterator}.
\item All other names use snake case\+: {\ttfamily unordered\+\_\+map}.
\end{DoxyItemize}\hypertarget{md_03-Style_autotoc_md3}{}\doxysection{\texorpdfstring{Distinguish Private Object Data}{Distinguish Private Object Data}}\label{md_03-Style_autotoc_md3}
Name private data with a {\ttfamily m\+\_\+} prefix to distinguish it from public data. {\ttfamily m\+\_\+} stands for "{}member"{} data.\hypertarget{md_03-Style_autotoc_md4}{}\doxysection{\texorpdfstring{Distinguish Function Parameters}{Distinguish Function Parameters}}\label{md_03-Style_autotoc_md4}
The most important thing is consistency within your codebase; this is one possibility to help with consistency.

Name function parameters with an {\ttfamily t\+\_\+} prefix. {\ttfamily t\+\_\+} can be thought of as "{}the"{}, but the meaning is arbitrary. The point is to distinguish function parameters from other variables in scope while giving us a consistent naming strategy.

Any prefix or postfix can be chosen for your organization. This is just one example. {\itshape This suggestion is controversial, for a discussion about it see issue \href{https://github.com/lefticus/cppbestpractices/issues/11}{\texttt{ \#11}}.}

Since a const member variable cannot be assigned a new value, such a class may not have a meaningful copy assignment operator.\hypertarget{md_03-Style_autotoc_md18}{}\doxysection{\texorpdfstring{Always Use Namespaces}{Always Use Namespaces}}\label{md_03-Style_autotoc_md18}
There is almost never a reason to declare an identifier in the global namespace. Instead, functions and classes should exist in an appropriately named namespace or in a class inside of a namespace. Identifiers which are placed in the global namespace risk conflicting with identifiers from other libraries (mostly \doxylink{classC}{C}, which doesn\textquotesingle{}t have namespaces).\hypertarget{md_03-Style_autotoc_md19}{}\doxysection{\texorpdfstring{Use the Correct Integer Type for Standard Library Features}{Use the Correct Integer Type for Standard Library Features}}\label{md_03-Style_autotoc_md19}
The standard library generally uses {\ttfamily std\+::size\+\_\+t} for anything related to size. The size of {\ttfamily size\+\_\+t} is implementation defined.

In general, using {\ttfamily auto} will avoid most of these issues, but not all.

Make sure you stick with the correct integer types and remain consistent with the C++ standard library. It might not warn on the platform you are currently using, but it probably will when you change platforms.

Ultimately this is a matter of preference, but .hpp and .cpp are widely recognized by various editors and tools. So the choice is pragmatic. Specifically, Visual Studio only automatically recognizes .cpp and .cxx for C++ files, and Vim doesn\textquotesingle{}t necessarily recognize .cc as a C++ file.

One particularly large project (\href{https://github.com/NREL/OpenStudio}{\texttt{ Open\+Studio}}) uses .hpp and .cpp for user-\/generated files and .hxx and .cxx for tool-\/generated files. Both are well recognized and having the distinction is helpful.\hypertarget{md_03-Style_autotoc_md21}{}\doxysection{\texorpdfstring{Never Mix Tabs and Spaces}{Never Mix Tabs and Spaces}}\label{md_03-Style_autotoc_md21}

Configure your editor so this does not happen.\hypertarget{md_03-Style_autotoc_md22}{}\doxysection{\texorpdfstring{Never Put Code with Side Effects Inside an assert()}{Never Put Code with Side Effects Inside an assert()}}\label{md_03-Style_autotoc_md22}

They can help you stick to \href{http://en.wikipedia.org/wiki/Don\%27t_repeat_yourself}{\texttt{ DRY principles}}. They should be preferred to macros, because macros do not honor namespaces, etc.\hypertarget{md_03-Style_autotoc_md24}{}\doxysection{\texorpdfstring{Use Operator Overloads Judiciously}{Use Operator Overloads Judiciously}}\label{md_03-Style_autotoc_md24}
Operator overloading was invented to enable expressive syntax. Expressive in the sense that adding two big integers looks like {\ttfamily a + b} and not {\ttfamily a.\+add(b)}. Another common example is {\ttfamily std\+::string}, where it is very common to concatenate two strings with {\ttfamily string1 + string2}.

However, you can easily create unreadable expressions using too much or wrong operator overloading. When overloading operators, there are three basic rules to follow as described \href{http://stackoverflow.com/questions/4421706/operator-overloading/4421708\#4421708}{\texttt{ on stackoverflow}}.

Specifically, you should keep these things in mind\+:


\begin{DoxyItemize}
\item Overloading {\ttfamily operator=()} when handling resources is a must. See \href{03-Style.md\#consider-the-rule-of-zero}{\texttt{ Consider the Rule of Zero}} below.
\item For all other operators, only overload them when they are used in a context that is commonly connected to these operators. Typical scenarios are concatenating things with +, negating expressions that can be considered "{}true"{} or "{}false"{}, etc.
\item Always be aware of the \href{http://en.cppreference.com/w/cpp/language/operator_precedence}{\texttt{ operator precedence}} and try to circumvent unintuitive constructs.
\item Do not overload exotic operators such as \texorpdfstring{$\sim$}{\string~} or \% unless implementing a numeric type or following a well recognized syntax in specific domain.
\item \href{http://stackoverflow.com/questions/5602112/when-to-overload-the-comma-operator?answertab=votes\#tab-top}{\texttt{ Never}} overload {\ttfamily operator,()} (the comma operator).
\item Use non-\/member functions {\ttfamily operator\texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}()} and {\ttfamily operator\texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}()} when dealing with streams. For example, you can overload {\ttfamily operator\texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}(std\+::ostream \&, My\+Class const \&)} to enable "{}writing"{} your class into a stream, such as {\ttfamily std\+::cout} or an {\ttfamily std\+::fstream} or {\ttfamily std\+::stringstream}. The latter is often used to create a string representation of a value.
\item There are more common operators to overload \href{http://stackoverflow.com/questions/4421706/operator-overloading?answertab=votes\#tab-top}{\texttt{ described here}}.
\end{DoxyItemize}

More tips regarding the implementation details of your custom operators can be found \href{http://courses.cms.caltech.edu/cs11/material/cpp/donnie/cpp-ops.html}{\texttt{ here}}.\hypertarget{md_03-Style_autotoc_md25}{}\doxysection{\texorpdfstring{Avoid Implicit Conversions}{Avoid Implicit Conversions}}\label{md_03-Style_autotoc_md25}
\hypertarget{md_03-Style_autotoc_md26}{}\doxysubsection{\texorpdfstring{Single Parameter Constructors}{Single Parameter Constructors}}\label{md_03-Style_autotoc_md26}
Single parameter constructors can be applied at compile time to automatically convert between types. This is handy for things like {\ttfamily std\+::string(const char \texorpdfstring{$\ast$}{*})} but should be avoided in general because they can add to accidental runtime overhead.

Instead mark single parameter constructors as {\ttfamily explicit}, which requires them to be explicitly called.\hypertarget{md_03-Style_autotoc_md27}{}\doxysubsection{\texorpdfstring{Conversion Operators}{Conversion Operators}}\label{md_03-Style_autotoc_md27}
Similarly to single parameter constructors, conversion operators can be called by the compiler and introduce unexpected overhead. They should also be marked as {\ttfamily explicit}.

\hypertarget{md_03-Style_autotoc_md28}{}\doxysection{\texorpdfstring{Consider the Rule of Zero}{Consider the Rule of Zero}}\label{md_03-Style_autotoc_md28}
The Rule of Zero states that you do not provide any of the functions that the compiler can provide (copy constructor, copy assignment operator, move constructor, move assignment operator, destructor) unless the class you are constructing does some novel form of ownership.

The goal is to let the compiler provide optimal versions that are automatically maintained when more member variables are added.

\href{http://www.nirfriedman.com/2015/06/27/cpp-rule-of-zero/}{\texttt{ This article}} provides a background and explains techniques for implementing nearly 100\% of the time.
